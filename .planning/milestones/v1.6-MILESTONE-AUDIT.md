## Integration Findings

### BLOCKER — MC6 controller.apply() result discarded; state_modified never set via controller path

**File:** `packages/rig/src/rig/engine/apply.py` lines 190–192
**Requirement:** MC6-01 ↔ STATE-01 (cross-phase wiring)

```python
controller.apply(controller_ctx)          # return value discarded
if controller.id in state.devices:        # always False on cold start
    state_modified = True
```

`MC6Device.apply()` returns `DeviceApplyResult` that is silently discarded. `state_modified` is only set `True` if `controller.id` is already in `state.devices` — which is `False` on the first run. If a scene apply runs only MC6 wiring changes with no device-action changes, `state_writer.write()` is never called and MC6 programming is not persisted to `state.json`. This is the same BLOCKER flagged in the 2026-06-23 audit and remains unresolved.

**Fix:** Capture the return value (`result = controller.apply(controller_ctx)`) and set `state_modified = True` when `result.status == "confirmed"`.

### WARNING — Catalog constants used in tests only; not in runtime engine

**Requirement:** CAT-01 (Phase 29)
**Files:** `packages/rig/src/rig/engine/plan/compute.py`, `packages/rig-chasebliss/src/rig_chasebliss/catalog.py`

`MOOD_MKII_WET_BYPASS_CC` and `MOOD_MKII_LOOP_BYPASS_CC` constants are used in catalog internal definitions and test assertions — but `_compute_param_diff` in `compute.py` derives CC numbers from YAML preset `parameters` values, not catalog constants. CAT-01 is satisfied (tests reference constants, no bare literals), but runtime divergence is possible if YAML and catalog constants ever disagree.
