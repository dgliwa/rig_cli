# Plan: 21-01 — Concrete Types at the Plugin Boundary

## Goal
Every plugin device class carries a concrete, named config type and a typed preset
list. No `config: Any` or `presets: list[Any]` remains in plugin source. A `Preset`
Protocol is defined in `rig.engine.plugin` and the Device Protocol references it.

## Success Criteria (from ROADMAP.md)
1. Each plugin (`ChaseBlissDevice`, `HXStompDevice`, `AnalogDevice`, `MC6Device`)
   declares `config: <ConcreteConfig>` — no `config: Any` in plugin source
2. A `Preset` Protocol is defined in `rig.engine.plugin` with the minimal interface
   the engine requires (`id: str`, `name: str`)
3. Each plugin device class declares `presets: list[<ConcretePreset>]` — no `list[Any]`
   in plugin preset fields
4. YAML construction validates against the concrete config type — invalid type for a
   known field raises `ValidationError`, not `AttributeError`

---

## Context: YAML config type dispatch

The plugin registry dispatches on `config.type` from the YAML. The keys are:
- `manual` → `AnalogDevice`
- `midi` → `HXStompDevice`
- `chase_bliss` → `ChaseBlissDevice`
- `controller` → `MC6Device`

All new config models include a matching `type` Literal to self-document and allow
config parsing to be self-describing.

### T6 — Fix plugin registry placeholder creation

**File:** `packages/rig/src/rig/engine/plugin_registry.py`

The current `_discover()` creates placeholder device instances via
`model_class(id=..., config=None)` then a fallback `model_class(id=..., config=object())`.
With concrete config types these will fail Pydantic validation.

Replace both `try/except` blocks with a single `model_construct` call:
```python
placeholder = model_class.model_construct(
    id=f"__{ep.name}__",
    name=str(ep.name),
    config=None,
    presets=[],
)
registry.register(ep.name, placeholder)
```
`model_construct` bypasses validators and sets fields directly — safe for a placeholder
that is never used for production data, only for Protocol-shape checks in tests.

**Verification:**
- `test_discovery_placeholder_implements_device_protocol` still passes
- `test_discovery_finds_registered_plugins` still passes
- No `ValidationError` during registry discovery with any of the 4 plugins

## Risk Notes

- **Pydantic dict coercion**: Pydantic v2 coerces `{"type": "manual"}` → `AnalogConfig`
  when a field is typed `AnalogConfig`. Passing a raw dict from YAML is fine — Pydantic
  validates and constructs the model. This is desired.
- **`model_construct` bypass**: Placeholder instances skip validators. They must never
  flow into production code paths — test-only, Protocol-shape checks only.
- **`from __future__ import annotations`**: All plugin files use postponed annotations.
  Top-level imports of concrete types (like `DigitalPreset`) must be at module scope,
  not inside `TYPE_CHECKING`, to be available at Pydantic model construction time.
- **`hasattr` guards in `apply.py`**: Kept intentionally because `Rig.devices: dict[str, Any]`
  is untyped until Phase 22. Removing these guards without closing that gap would risk
  `AttributeError` on malformed device objects.
- **MC6 `scenes` field**: `MC6Config.scenes` stores the scene definitions that
  `Rig.scenes` reads via `getattr(cfg, "scenes", {})`. The guard in `Rig.scenes` is
  kept for Phase 22 to clean up.
