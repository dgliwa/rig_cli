<objective>
Add `apply_device_preset()` to the engine and wire `--device`/`--preset` flags into the CLI,
enabling `rig apply --device <id> --preset <id>` to apply exactly one device's preset without
a scene-based plan. All other devices are skipped. Only that device's state.json entry is
updated; state.scenes is left untouched.

Purpose: Mid-session targeted reapply and one-off preset sends without re-running the full
scene apply flow.

Output:
- New `apply_device_preset()` function in `packages/rig/src/rig/engine/apply.py`
- `--device` and `--preset` Typer options with mutual-exclusion validation in
  `packages/rig/src/rig/cli/commands/apply.py`
- Unit tests in `packages/rig/tests/test_apply_device_preset.py`
- CLI-level validation tests in `packages/rig/tests/test_cli_apply.py`
</objective>

      def apply_device_preset(
          device_id: str,
          preset_id: str,
          *,
          state_writer: StateWriter,
          confirmation_io: ConfirmationIO | None = None,
          rig: Rig,
          config_path: str | None = None,
          dry_run: bool = False,
          midi: MidiManager | None = None,
      ) -> DeviceApplyResult:

    Implementation steps (per D-01 through D-07 — do NOT deviate):
    1. Resolve device: `device = rig.devices[device_id]` (caller already validated)
    2. Read state: `state = state_writer.read(config_path) if config_path else RigState()`
    3. Build `connected_devices: set[str] = set()`
    4. Resolve IO: `_io = confirmation_io or RichConfirmationIO()`
    5. Build SetupContext (same fields as apply_plan uses) and call `setup_result = device.setup(setup_ctx)`.

    7. Resolve midi_channel: `getattr(device.config, "midi_channel", None)`
    8. Build `DeviceAction` with `status=ActionStatus.CONFIGURE` (import from `rig.engine.plan.models`).
       Always use CONFIGURE — no short-circuit for "already applied" in this path (per RESEARCH.md rationale)
    9. Build DeviceApplyContext with the action and call `result = device.apply(device_ctx)`
    10. If `result.status == "confirmed"` and not `dry_run` and `config_path`:
        call `state_writer.write(config_path, state)`.
        Do NOT call update_device_state() — that is already done inside device.apply() per D-07.
        Do NOT touch state.scenes.
    11. Return result

    Run tests (GREEN). Commit: `feat(apply): add apply_device_preset for isolated single-device apply`

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-26-01 | Spoofing | `--device` / `--preset` CLI args | mitigate | Validate both IDs against loaded `Rig.devices` and `device.presets` before MIDI — unknown IDs exit 1 immediately (V5 ASVS L1) |

| T-26-03 | Elevation of Privilege | MIDI port opened before validation | mitigate | All validation (flag logic, device/preset existence) runs after `load_rig()` and before `MidiManager()` construction — no port opened on validation failure |
</threat_model>
